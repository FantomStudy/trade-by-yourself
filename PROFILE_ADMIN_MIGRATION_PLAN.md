# План Рефакторинга И Переноса `profile` / `admin`

## Зачем Этот Документ

Этот файл нужен как рабочая карта на ближайшие дни, чтобы перенос `profile` и `admin` не превратился в хаотичное возвращение legacy-кода под новые пути.

Цель:

- довести рефакторинг до простого и читаемого состояния
- сохранить минимализм в духе antfu
- не тащить обратно старые абстракции ради скорости
- двигаться маленькими, безопасными шагами

Главная идея:

- переносить не папки целиком, а только полезную логику и UI
- сначала выстраивать понятные API-границы
- потом переносить конкретную страницу
- всё сложное и связанное откладывать до тех пор, пока не будет ясной основы

---

## Что Уже Происходит В Репозитории

По текущему `git` видно, что рефакторинг уже начат и направление в целом правильное:

- публичная часть выносится в route groups, например `src/app/(public)`
- часть `profile` уже живая и находится в `src/app/profile`
- старые страницы `profile` и `admin` убраны из активного роутинга и лежат в `src/app/_excluded/...`
- старый API-слой и старые хуки постепенно убираются

Это хороший знак. Значит, не нужно "возвращать всё назад". Нужно аккуратно закончить переход.

## Где Сейчас Основная Проблема

Сейчас в проекте есть фактически две параллельные реальности:

1. Новая:
   - `src/app/profile`
   - `src/app/(public)`
   - явные `src/api/*`

2. Старая:
   - `src/app/_excluded/profile`
   - `src/app/_excluded/admin`
   - legacy-импорты из `@/api/hooks`, `@/api/requests`, `@/lib/contexts`, `@/lib/support-chat`

Главный риск в том, что при переносе легко случайно восстановить старую архитектуру:

- вернуть глобальные generic hooks
- вернуть общий `requests/index`
- снова размазать типы и запросы по нескольким уровням
- притащить `_excluded` как источник зависимостей

Этого делать нельзя.

## Базовые Принципы Переноса

### 1. `_excluded` — это архив, а не рабочий слой

Разрешённое использование:

- открыть файл
- понять логику
- взять кусок UI
- переписать под новую архитектуру

Неразрешённое использование:

- импортировать что-то из `src/app/_excluded/*` в живой код
- переносить страницу целиком вместе со всеми старыми зависимостями

### 2. Явные доменные модули лучше общего слоя

Плохо:

- `@/api/hooks`
- `@/api/requests`
- "универсальные" mutation/query-обёртки на весь проект

Хорошо:

- `src/api/users/getUser.ts`
- `src/api/users/updateSettings.ts`
- `src/api/payments/createPayment.ts`
- `src/api/chats/getChats.ts`
- `src/api/reviews/moderateReview.ts`

То есть каждый импорт должен сам объяснять, что именно происходит.

### 3. Серверные страницы по умолчанию

Если страница просто читает данные и рендерит их:

- сначала пробуем server component

Если на странице форма, диалог, локальный state, drag/drop, file upload:

- оставляем маленький client island

Не нужно делать client page только потому, что так было раньше.

### 4. Выносить общее только после второй реальной повторяемости

Если компонент нужен только одной странице:

- он должен жить рядом с ней

Если компонент реально нужен и `profile`, и `admin`, и API/поведение совпадают:

- тогда уже можно вынести в `src/components/*`

Это важно, чтобы не размножать "shared" папки раньше времени.

### 5. Каждый перенос должен уменьшать сложность

После переноса страница должна:

- иметь меньше скрытых зависимостей
- меньше опираться на старые хуки
- читаться быстрее, чем раньше

Если после переноса она стала только длиннее и запутаннее, значит перенос пошёл не туда.

## Общая Стратегия: Волны Переноса

Ниже порядок, который даст максимум прогресса при минимальном риске.

### Важная Поправка По Приоритетам

`create/edit product` нужно считать не "уже почти готовым", а отдельным незавершённым центром `profile`.

Причина:

- это главный сценарий кабинета продавца
- новые маршруты уже заведены, но этого недостаточно
- если не добить форму товара сейчас, дальше начнётся дублирование логики между старым и новым миром

Что считать завершением этой задачи:

- `src/app/profile/products/create/page.tsx`
- `src/app/profile/products/edit/[productId]/page.tsx`
- общая форма товара
- единый поток данных для create/edit
- отсутствие зависимости на legacy `src/app/_excluded/profile/create-product/page.tsx`

Рекомендуемая форма решения:

- одна общая route-local форма товара
- `create` задаёт режим создания
- `edit` задаёт режим редактирования и подгружает initial data
- API остаётся явным:
  - `src/api/products/createProduct.ts`
  - `src/api/products/updateProduct.ts`
  - `src/api/products/getProduct.ts`
  - `src/api/categories/*`
  - `src/api/address/*` при необходимости

Итог:

- сначала добить create/edit product
- потом уже переносить остальные "простые" страницы `profile`

## Волна 1. Добить Простые Страницы `profile`

### Что переносить

- `src/app/_excluded/profile/settings/page.tsx`
- `src/app/_excluded/profile/banner-request/page.tsx`
- `src/app/_excluded/profile/banner-stats/page.tsx`
- `src/app/_excluded/profile/promotion-request/page.tsx`

### Почему именно они

- они локальны
- не формируют системную архитектуру проекта
- не требуют сразу решать чат, сокеты, модерацию или платёжный пайплайн
- дают быстрый результат и закрывают видимую часть кабинета

### Что делать при переносе

Для каждой страницы:

1. создать настоящий route folder в `src/app/profile/...`
2. перенести только UI-каркас и полезные подкомпоненты
3. сразу выкинуть legacy-импорты
4. заменить их на явные `src/api/*`
5. если нужного API ещё нет, сначала создать маленький explicit function

### Что в итоге должно получиться

После этой волны у `profile` появится почти весь понятный пользовательский кабинет, кроме сложных зон:

- messages
- analytics
- balance

Это уже даст ощущение завершённой структуры.

## Волна 2. Закрыть `analytics` и `balance`

### Что переносить

- `src/app/_excluded/profile/analytics/page.tsx`
- `src/app/_excluded/profile/balance/page.tsx`

### Почему не раньше

Обе страницы тянут за собой старые зависимости:

- deprecated hooks
- react-query слой старого образца
- неявные payment/analytics контракты

Если переносить их слишком рано, велик шанс восстановить старую архитектуру ради скорости.

### Что нужно сделать до переноса

Для `analytics`:

- понять, какие метрики реально нужны
- не тащить `useAnalytics` из deprecated-зоны
- собрать минимальный новый контракт под нужные данные

Для `balance`:

- выделить отдельный payment-домен
- убрать зависимость от старых payment hooks
- сделать явные функции:
  - `getPaymentHistory`
  - `createPayment`
  - `checkPaymentStatus`

### Рекомендуемая форма

`analytics`:

- серверная страница
- маленький client filter, только если реально нужен интерактивный выбор периода

`balance`:

- server data + отдельный client component для пополнения и проверки статуса

## Волна 3. Перенос `profile/messages`

### Что переносить

- `src/app/_excluded/profile/messages/page.tsx`
- `src/app/_excluded/profile/messages/[chatId]/page.tsx`
- `src/app/_excluded/profile/messages/support/page.tsx`

### Почему это отдельная волна

Это не просто страницы. Это кусок архитектуры:

- чаты
- сообщения
- unread counters
- support-логика
- возможно, сокеты
- возможно, локальное состояние и псевдо-backend

### Главный вопрос до начала переноса

Нужно принять одно решение:

- `support` — это часть настоящей chat-системы
- или временная локальная/псевдо-система, но изолированная через один adapter

Пока этого решения нет, переносить UI бессмысленно: получится новый путь со старым хаосом.

### Как правильно переносить

Сначала:

- спроектировать `src/api/chats/*`
- определить модель данных
- решить судьбу `support-chat`
- решить судьбу `lib/contexts/chat`

Потом:

- переносить список чатов
- потом детальную страницу чата
- потом support

### Почему support надо делать вместе с messages

Потому что support — это не отдельный виджет, а ещё один сценарий общения. Если делать его отдельно, велика вероятность получить два разных механизма сообщений.

## Волна 4. Поднять Каркас `admin`

### Что переносить

- `src/app/_excluded/admin/layout.tsx`
- `src/app/_excluded/admin/page.tsx`
- `src/app/_excluded/admin/_components/admin-sidebar.tsx`

### Почему это нужно сделать рано

У `admin` должен появиться стабильный позвоночник:

- layout
- guard на admin-доступ
- sidebar
- redirect с `/admin`

Без этого каждая следующая страница будет переноситься в подвешенное состояние.

### Что улучшить прямо в момент переноса

- переименовать `admin-sidebar` в `AdminSidebar`
- вынести меню в `constants.ts`
- оставить state/provider рядом с sidebar
- не выносить это в глобальные shared/dashboard абстракции

### Что важно не сломать

- проверка прав через `checkIsAdmin`
- логика `notFound()` / guard
- мобильное поведение sidebar

## Волна 5. Простые и Средние Разделы `admin`

### Что переносить

- `categories`
- `advertising`
- `users`
- `promotion`
- `logs`

### Почему именно в таком наборе

Они ещё не простые, но уже достаточно локальные:

- есть ограниченная предметная область
- нет такой глубокой связанности, как у `support`
- нет такой архитектурной тяжести, как у chat-flow

### Подход по каждому разделу

`categories`

- хороший кандидат на ранний перенос
- скорее всего уместится в explicit `src/api/categories/*`
- UI можно оставить client-side, если там много интерактива

`advertising`

- переносить после или вместе с `categories`
- заменить остатки `@/api/requests/banner` на `src/api/banners/*`
- не плодить отдельный старый requests-слой ради нескольких действий

`users`

- требует аккуратности
- сейчас завязан на `@/api/requests`
- нужно вынести admin-actions в `src/api/users/*`

Пример:

- `findAllUsers.ts`
- `updateUser.ts`
- `toggleBanned.ts`
- `deleteUser.ts`

`promotion`

- здесь важно заранее решить, promotion — это отдельный домен или часть products
- если это самостоятельная сущность, лучше `src/api/promotions/*`
- если это строго надстройка над товаром, можно жить под products

Главное — выбрать один путь и не дублировать.

`logs`

- отдельный маленький домен
- при необходимости создать `src/api/logs/*`
- не прятать логи в общем "admin requests index"

## Волна 6. `admin/moderation` и `admin/reviews`

### Что переносить

- `src/app/_excluded/admin/moderation/page.tsx`
- `src/app/_excluded/admin/reviews/page.tsx`

### Почему это поздняя волна

Потому что это уже не просто CRUD:

- есть workflow
- есть статусы
- есть approve / reject сценарии
- есть modal flows
- есть pagination и фильтрация

Если перенести их слишком рано, легко восстановить старые generic mutation hooks.

### Как переносить правильно

Не тащить обратно:

- `useModerateProductMutation`
- `useModerationProduct`
- `useProductsToModerate`
- `useModerateReviewMutation`
- `useReviewsToModerate`

Вместо этого создать явные функции:

- `src/api/products/getProductsToModerate.ts`
- `src/api/products/getModerationProduct.ts`
- `src/api/products/moderateProduct.ts`
- `src/api/reviews/getReviewsToModerate.ts`
- `src/api/reviews/moderateReview.ts`

Если потом нужны query hooks:

- класть их рядом с функцией
- а не возвращать глобальный `api/hooks`

### Как раскладывать UI

Оставить крупные куски рядом со страницей:

- `ProductDetailDialog`
- `RejectModal`
- `ModerationListItem`
- `ModerationFilters`

Это лучше, чем преждевременно тащить всё в `components/admin`.

## Волна 7. `admin/support`

### Что переносить

- `src/app/_excluded/admin/support/page.tsx`

### Почему это должен быть самый последний перенос

Потому что `support` связан с той же проблемой, что и `profile/messages`:

- старый chat context
- старая support-state логика
- неочевидный источник истины

Если заняться `admin/support` раньше `messages`, получится перекошенная архитектура:

- одна система для user chat
- другая система для support
- третья система для unread или live updates

Правильнее рассматривать `profile/messages` + `admin/support` как одну архитектурную задачу.

### Что важно

- не размазывать бизнес-логику по UI-обработчикам
- не переносить консольные логи
- не восстанавливать старые generic mutation hooks ради скорости

## Какие Legacy-Слои Не Нужно Возвращать

Ниже список того, что лучше не реанимировать:

- `src/api/hooks/*`
- `src/api/requests/*`
- `src/app/_excluded/api/*`
- `src/hooks/useUser.ts`
- `src/hooks/useUserProducts.ts`
- старые chat contexts в `src/lib/contexts/*`

Допустим только один компромисс:

- если какой-то hook нужен временно, он должен лежать рядом с конкретной explicit API-функцией

Пример:

```text
src/api/reviews/getReviewsToModerate.ts
src/api/reviews/useReviewsToModerate.ts
```

А не:

```text
src/api/hooks/queries/useReviewsToModerate.ts
```

## Правила Очистки Во Время Переноса

На каждой странице по дороге нужно убирать мусор:

- `console.log`
- мёртвые комментарии
- старые импорты, которые тянут за собой половину legacy-слоя
- `as any`, если исправление недорогое
- inline styles, если локальный CSS module или существующие utility-классы читаются лучше

При этом не нужно:

- редизайнить весь UI
- переписывать все стили проекта разом
- придумывать новый design system слой
- переносить unrelated public-код под шумок

Задача сейчас — архитектурная чистка и перенос, а не большой редизайн.

## Мини-Чеклист Перед Переносом Любой Страницы

- Есть ли у страницы явный API-модуль?
- Можно ли сделать её server-first?
- Не тянет ли она старые hooks/requests?
- Нужен ли shared component на самом деле?
- Не решаю ли я сейчас две задачи сразу, хотя можно решить одну?

---

## Мини-Чеклист После Переноса

- Нет импортов из `_excluded`
- Нет новых импортов из старого `requests/hooks` слоя
- Путь реально открывается через app router
- Типы лежат рядом с доменом
- Страницу можно понять без путешествия по пяти старым каталогам

---

## Когда Надо Остановиться И Переосмыслить Подход

Если в процессе появится хоть один из этих симптомов, лучше остановиться:

- снова хочется восстановить общий `src/api/hooks`
- `_excluded` начинает использоваться как библиотека
- появляется идея сделать общий `dashboard-core` для `profile` и `admin`
- chat/support требуют всё больше исключений и костылей
- ради одного экрана приходится тащить старый слой целиком

Это почти всегда сигнал, что перенос идёт не к упрощению, а к консервации legacy.

---

## Определение Готовности

Работу можно считать завершённой, когда:

- все рабочие `profile` route лежат в `src/app/profile`
- все рабочие `admin` route лежат в `src/app/admin`
- `src/app/_excluded/profile` и `src/app/_excluded/admin` больше не нужны
- старые `requests/hooks/contexts` либо удалены, либо сведены к крошечным адаптерам
- структура проекта читается сверху вниз без поиска "где же тут настоящий источник данных"

## Самая Короткая Версия

Если завтра нужен совсем короткий стартовый маршрут:

1. Сначала закончить:
   - `profile/products/create`
   - `profile/products/edit/[productId]`
   - общую форму товара для create/edit
2. Потом закончить:
   - `profile/settings`
   - `profile/banner-request`
   - `profile/banner-stats`
   - `profile/promotion-request`
3. Потом поднять:
   - `admin/layout`
   - `admin/sidebar`
   - `admin/categories`
   - `admin/advertising`
4. Не трогать пока:
   - `messages`
   - `support`
   - `balance`
   - `moderation`
   - `reviews`
5. Не переносить legacy hooks целиком.
6. Для каждой страницы сначала создавать явный API-слой, потом переносить UI.
