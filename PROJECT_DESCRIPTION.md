# Описание проекта «Торгуй сам»

## 1. Общие сведения о проекте

**Название проекта:** «Торгуй сам» (внутреннее имя пакета: `next-commerce-tailwind`)

**Тип приложения:** Веб-приложение — онлайн-маркетплейс (доска объявлений) для размещения и поиска товаров, аналог Avito/Юла.

**Назначение:** Платформа предназначена для размещения объявлений о продаже товаров частными лицами, индивидуальными предпринимателями и юридическими лицами (ООО). Пользователи могут публиковать товары, искать и фильтровать объявления, вести переписку с продавцами в реальном времени, оставлять отзывы, управлять балансом и продвигать свои объявления.

**Версия:** 0.1.0 (стадия активной разработки)

**Локализация:** Русский язык (интерфейс полностью на русском, поддержка российских номеров телефона в формате +7)

## 3. Архитектура приложения

### 3.1. Общая архитектура

Приложение построено по клиент-серверной архитектуре:

- **Frontend (данный проект)** — Next.js 16 приложение с использованием App Router, гибридный рендеринг (SSR + CSR)
- **Backend (внешний сервис)** — REST API, к которому фронтенд обращается через HTTP-клиент `ofetch`. URL бэкенда задаётся переменной окружения `NEXT_PUBLIC_API_URL`
- **WebSocket-сервер** — для realtime-чатов, URL задаётся переменной `NEXT_PUBLIC_API_WS_URL`

### 3.3. Защита маршрутов (Middleware)

Файл `middleware.ts` реализует серверную защиту маршрутов:

- **Защищённые префиксы:** `/profile`, `/admin`
- **Механизм:** Проверка наличия cookie `session_id`
- **Поведение при отсутствии сессии:** Редирект на `/` с query-параметром `?auth=1` (для автоматического открытия диалога авторизации)

Дополнительная проверка для администраторов: в `admin/layout.tsx` выполняется серверный запрос `checkIsAdmin()` к API, и при отсутствии прав доступа пользователь перенаправляется на страницу 404.

```
    ├── banner.ts        # Типы баннеров
    └── review.ts        # Типы отзывов и модерации
```

**HTTP-клиент (ofetch):**
- Базовый URL из переменной окружения `NEXT_PUBLIC_API_URL`
- Автоматическая отправка cookies (`credentials: "include"`)
- SSR-поддержка: на сервере cookies пробрасываются из `next/headers`
- При ответе 401 на клиенте: автоматическая очистка cookie `session_id` и сброс кэша текущего пользователя

### 3.5. Провайдеры и контексты

Цепочка провайдеров в `providers.tsx`:

```
NuqsAdapter (синхронизация URL-состояния)
  └── QueryClientProvider (TanStack React Query)
       └── HydrationBoundary (SSR-гидрация данных)
            └── AuthProvider (контекст авторизации)
                 └── ChatSocketProvider (WebSocket для чатов)
```

**AuthProvider:**
- Выполняет запрос текущего пользователя через React Query
- Предоставляет объект `user` и функцию `logout`
- `logout` очищает cookie и сбрасывает кэш

**ChatSocketProvider:**
- Подключается к WebSocket-серверу при наличии авторизованного пользователя
- Обрабатывает входящие сообщения, обновляет список чатов
- Показывает push-уведомления через Web Notifications API
- Предоставляет объект `socket` и флаг `isConnected`

## 4. Функциональные модули

### 4.1. Модуль аутентификации

**Расположение:** `src/components/auth-dialog/`

**Механизм:** Сессионная аутентификация на основе cookie `session_id`.

**Экраны:**
1. **Вход (LoginScreen)** — авторизация по логину (email/телефон) и паролю
2. **Регистрация (RegisterScreen)** — регистрация с указанием ФИО, email, телефона и пароля. Поддержка верификации через SMS или Telegram
3. **Подтверждение кода (VerifyCodeScreen)** — ввод 6-значного SMS-кода для верификации номера телефона
4. **Восстановление пароля (RecoverScreen)** — восстановление доступа по email (отправка кода, ввод кода, смена пароля)

**Валидация форм:**
- Zod-схемы: `loginSchema`, `registerSchema`, `verifyMobileCodeSchema`, `forgotPasswordSchema`
- Валидация российских номеров в формате +7 (XXX) XXX-XX-XX
- Минимальная длина пароля — 6 символов

**API-эндпоинты:**
- `POST /auth/sign-in` — авторизация
- `POST /auth/sign-up?where=sms|telegram` — регистрация
- `POST /auth/verify-mobile-code` — подтверждение SMS-кода
- `POST /auth/forgot-password` — запрос на восстановление
- `POST /auth/verify-code` — подтверждение кода восстановления
- `POST /auth/change-password` — смена пароля
- `GET /auth/me` — получение текущего пользователя
- `GET /auth/isAdmin` — проверка прав администратора

### 4.2. Модуль каталога и поиска товаров

**Расположение:** `src/app/(feed)/(search)/`

**Главная страница** отображает ленту всех товаров с возможностью фильтрации, поиска и навигации по категориям.

**Иерархия каталога:**
1. **Категория** (например, «Электроника»)
2. **Подкатегория** (например, «Телефоны»)
3. **Тип** (например, «Смартфоны»)
4. **Характеристики типа** (например, «Объём памяти», «Цвет»)

**Поддержка двух подходов к фильтрации:**
- **ID-based** (числовые ID категорий)
- **Slug-based** (читаемые slug'и для SEO-дружественных URL, например `/elektronika/telefony/smartfony`)

**Параметры фильтрации:**
- Категория / подкатегория / тип (по slug или ID)
- Диапазон цены (minPrice, maxPrice)
- Состояние товара: NEW (новый) / USED (б/у)
- Тип продавца: INDIVIDUAL (физ. лицо) / IP / ООО
- Рейтинг продавца (minRating, maxRating, 1–5)
- Динамические характеристики (fieldValues — запись ключ-значение)
- Текстовый поиск (search)
- Регион
- Сортировка: по дате, по цене, по релевантности
- Пагинация (page, limit)

**Автоматическое динамическое получение доступных фильтров:**
- Эндпоинт `GET /product/available-filters` возвращает только актуальные значения (диапазон цен, состояния, типы продавцов, поля характеристик)

**Компоненты поиска:**
- `SearchBox` — поле текстового поиска
- `CategoryDialog` — диалог выбора категории
- `FiltersButton` — кнопка открытия панели фильтров
- `RegionButton` — выбор региона
- `Filters` — панель фильтров с динамическими полями

#### 4.7.1. Модерация товаров (`/admin/moderation`)

**Назначение:** Ревью объявлений, поданных пользователями, перед их появлением в публичной ленте.

**Механизм:**
- Загружает список товаров со статусом `MODERATE` через хук `useProductsToModerate` (`GET /product/moderation`)
- Отображает их в виде `FeedWrapper` — той же карточной ленты, что используется на публичных страницах

**Действия администратора:**
- **Одобрить** — отправляет `PATCH /product/moderate/:id` с `{ status: "APPROVED" }`, товар появляется в ленте
- **Отклонить** — открывает модальное окно (`Dialog`) с полем `Textarea` для ввода причины отказа, затем отправляет `{ status: "DENIED", reason: "..." }`. Причина сохраняется и видна пользователю в разделе «Мои объявления»

**Компоненты:**
- `useProductsToModerate` — React Query хук, запрашивает товары на модерации
- `useModerateProductMutation` — мутация для отправки решения
- `Dialog` + `Textarea` — форма указания причины отклонения
- `FeedWrapper` — компонент ленты карточек
- Кнопки: иконка `Check` (одобрить, зелёная) и иконка `X` (отклонить, красная)

**Состояния:**
- `moderatingId` — ID товара, по которому идёт операция (блокирует повторные клики)
- `showReasonModal` / `selectedProductId` / `rejectReason` — управление диалогом отклонения

#### 4.7.2. Модерация отзывов (`/admin/reviews`)

**Назначение:** Проверка отзывов о продавцах перед их публикацией.

**Данные:** Хук `useReviewsToModerate` (`GET /reviews/moderation`). Каждый отзыв содержит:
- Рейтинг (1–5 звёзд, отображается компонентом `Star`)
- Текст отзыва
- Информация об авторе (имя, ID)
- Информация о продавце, которому оставлен отзыв

**Действия:**
- **Одобрить** — `PATCH /reviews/moderate/:id` с `{ status: "APPROVED" }`
- **Отклонить** — открывает `Dialog` с подтверждением, затем отправляет `{ status: "REJECTED" }` через `useModerateReviewMutation`

**Компоненты:**
- `useReviewsToModerate` — React Query хук
- `useModerateReviewMutation` — мутация для решения по отзыву
- `Dialog` — диалог подтверждения отклонения
- Иконки `Check` и `X` для действий; `Star` для визуализации рейтинга

**Состояния:**
- `moderatingId` — ID обрабатываемого отзыва
- `showRejectDialog` / `selectedReviewId` — управление диалогом

#### 4.7.3. Управление категориями (`/admin/categories`)

**Назначение:** Полное управление иерархией каталога: Категория → Подкатегория → Тип → Характеристика (Поле).

**Иерархия и API-эндпоинты:**

| Уровень | GET (загрузка) | POST (создать) | PATCH (изменить) | DELETE (удалить) |
|---|---|---|---|---|
| Категория | `GET /category/find-all` | `POST /category/create` | `PATCH /category/update/:id` | `DELETE /category/delete/:id` |
| Подкатегория | `GET /subcategory/find-all` | `POST /subcategory/create` | `PATCH /subcategory/update/:id` | `DELETE /subcategory/delete/:id` |
| Тип | `GET /subcategory-type/find-all` | `POST /subcategory-type/create` | `PATCH /subcategory-type/update/:id` | `DELETE /subcategory-type/delete/:id` |
| Характеристика | `GET /type-field/find-all` | `POST /type-field/create` | `PATCH /type-field/update/:id` | `DELETE /type-field/delete/:id` |

Все 4 уровня загружаются параллельно через `Promise.all` при монтировании компонента.

**UI — древовидный интерфейс с раскрытием:**
- Категория — верхний уровень. Кнопка `ChevronRight/ChevronDown` раскрывает список подкатегорий
- Подкатегория — второй уровень. Аналогичная кнопка раскрывает типы
- Тип — третий уровень. Раскрывает список характеристик (type fields)
- Характеристика (Поле) — листовой уровень. Только редактирование и удаление

**Операции на каждом уровне:**
- **Добавить** — `Button` с `Plus` открывает модальный `Dialog` с полем `Input` (имя). При сохранении — POST-запрос, затем `loadData()` для обновления
- **Редактировать** — иконка `Edit` открывает диалог с заполненным `Input` (текущее имя). При сохранении — PATCH-запрос
- **Удалить** — иконка `Trash2` открывает диалог подтверждения. При подтверждении — DELETE-запрос
- При создании подкатегории/типа/характеристики — передаётся ID родительского элемента

**Состояния:**
- `expandedCategories`, `expandedSubcategories`, `expandedTypes` — `Set<number>` для контроля раскрытых узлов
- `showAdd*`, `showEdit*`, `editing*`, `*Name` — состояния модальных окон для каждого уровня
- `isSubmitting` — блокировка кнопок во время запроса
- `selectedSubcategoryId`, `selectedTypeId` — текущий родительский элемент при создании дочернего

**Локальные интерфейсы TypeScript:** `Category`, `Subcategory`, `SubcategoryType`, `TypeField` — определены прямо в файле страницы.

#### 4.7.4. Управление пользователями (`/admin/users`)

**Назначение:** Просмотр, поиск и управление всеми зарегистрированными пользователями.

**Компоненты:**
- `UsersPage` — страница с полем поиска и таблицей
- `UsersTable` (`_components/users-table.tsx`) — вынесен в отдельный компонент с собственным состоянием

**Загрузка данных:**
- `useQuery` с `queryFn: findAllUsers` (`GET /user/find-all`)
- Сортировка: по убыванию ID (новые пользователи сверху)

**Поиск (клиентская фильтрация по полю `searchQuery`):**
- По числовому ID (`user.id.toString().includes(query)`)
- По ФИО (`user.fullName.toLowerCase().includes(query)`)
- По Email (`user.email?.toLowerCase().includes(query)`)
- По номеру телефона (`user.phoneNumber.includes(query)`)

**Таблица пользователей — столбцы:**

| Столбец | Описание |
|---|---|
| ID | Числовой идентификатор |
| ФИО | Ссылка на страницу продавца (`/seller/:id`) |
| Email | Email-адрес |
| Телефон | Номер в формате +7 |
| Тип | INDIVIDUAL / IP / ООО |
| Баланс | Основной баланс в рублях |
| Бонусы | Бонусный баланс |
| Товары | Количество объявлений |
| Статус | «Заблокирован» / «Активен» с цветовой индикацией |
| Действия | Кнопки: редактировать, бан/разбан, удалить |

**Действия:**
- **Редактировать** (иконка `Edit`) — открывает `Dialog` с формой редактирования полей: `fullName`, `email`, `phoneNumber`, `profileType`, `bonusBalance`. Мутация `updateUser(id, data)` (`PATCH /user/update/:id`)
- **Бан / Разбан** (иконка `Ban`) — переключает статус блокировки через `toggleUserBanned(id)` (`PATCH /user/ban/:id`). Заблокированный пользователь не может входить на сайт
- **Удалить** (иконка `Trash2`) — открывает диалог подтверждения, затем `deleteUser(id)` (`DELETE /user/delete/:id`)

**React Query мутации:** `updateMutation`, `deleteMutation`, `toggleBanMutation` — все инвалидируют кэш `["admin-users"]` после успешного выполнения.

#### 4.7.5. Системные логи (`/admin/logs`)

**Назначение:** Мониторинг действий пользователей и отладка.

- `id` — ID записи лога
- `userId` — ID пользователя, совершившего действие

- По `log.user.fullName` (без учёта регистра)
- По `log.user.email`
- По `log.user.id`

**Сортировка:** По ID в обратном порядке — новые записи сверху (`sort((a, b) => b.id - a.id)`).

#### 4.7.6. Управление рекламой (`/admin/advertising`)

**Назначение:** Создание, обновление и удаление рекламных баннеров, размещаемых на сайте.

**Компонент `BannerSelector` (`_components/banner-selector.tsx`):**
Загружает существующие баннеры через `getBanners()` (`GET /banner`) и рендерит карточку для каждого из 4 слотов размещения.

**Конфигурация слотов (`BANNER_CONFIGS`):**

| Слот (`place`) | Размер | Описание |
|---|---|---|
| `PRODUCT_FEED` | 320 × 400 px | Баннер в ленте карточек (вертикальный) |
| `PROFILE` | 660 × 400 px | Большой баннер в ленте карточек |
| `CHATS` | 1280 × 400 px | Баннер среди чатов |
| `FAVORITES` | 1280 × 200 px | Баннер под хедером на главной странице |

**Поля формы для каждого баннера:**
- **Название баннера** (`Input`) — сохраняется в `bannerNames[itemId]`
- **URL перехода** (`Input`) — сохраняется в `bannerUrls[itemId]`, валидируется через `new URL()`. Открывается при клике на баннер
- **Файл изображения** — скрытый `<input type="file">` с `ref`, доступный через `fileInputRefs`

**Операции:**
- **Загрузить / Обновить** — если для данного слота уже существует баннер (`bannerId`), вызывает `updateBanner(id, { image, name, navigateToUrl })` (`PATCH /banner/:id`); иначе — `createBanner({ image, name, navigateToUrl, place })` (`POST /banner/create`)
- **Удалить** — `deleteBanner(id)` (`DELETE /banner/:id`), возвращает слот в пустое состояние

**Предпросмотр:** `URL.createObjectURL(file)` для создания локального превью выбранного изображения до загрузки.

**Состояния:** `banners`, `loading`, `uploadingId` (блокирует кнопку загрузки), `previewImages`, `bannerNames`, `bannerUrls`, `fileInputRefs`.

#### 4.7.8. Техподдержка (`/admin/support`)

**Назначение:** Мониторинг всех чатов пользователей платформы и ответы в чате технической поддержки.

**Два режима работы на одной странице:**

**1. Список чатов пользователей (основной режим)**
- Загружает все чаты через хук `useChats` (`GET /chat/all`)
- Отображает карточки чатов: аватар участников, тема (название товара), последнее сообщение, время активности, счётчик непрочитанных
- Сортировка: по `lastActivity` (актуальные — сверху)
- Ссылки ведут на страницу конкретного чата через `Link` (для просмотра переписки)
- WebSocket (`useChatSocket`) обновляет список в реальном времени: обработчик `socket.on("newMessage", ...)` обновляет `lastMessage`, `unreadCount`, `lastActivity` нужного чата без перезагрузки страницы

**2. Чат технической поддержки**
- Отдельная секция (кнопка с иконкой `MessageSquare` и бейджем непрочитанных)
- Хранилище: `localStorage` (без серверной части) через утилиты из `src/lib/support-chat.ts`:
  - `getSupportMessages()` — все сообщения
  - `addSupportMessage(text, isFromUser)` — добавить сообщение
  - `markSupportMessagesAsRead(isUser)` — сбросить счётчик непрочитанных
  - `getAdminUnreadCount()` — количество непрочитанных у администратора
- Интерфейс чата: область сообщений с автопрокруткой (`messagesEndRef`), поле ввода + кнопка отправки (`Send`)
- Отправка по `Enter` (без Shift) или кнопке
- При открытии чата — автоматически читает непрочитанные (`markSupportMessagesAsRead(false)`)
- События `supportMessageAdded` и `supportMessagesRead` (CustomEvent в window) синхронизируют состояние между вкладками

**Состояния:**
- `chats` — текущий список чатов (обновляется через WebSocket)
- `showSupportChat` — переключение между видом списка и чатом поддержки
- `supportMessages`, `supportUnread` — данные из localStorage
- `message` — черновик сообщения в чате поддержки

### 4.8. Модуль чата (Realtime)

**Технология:** Socket.IO (WebSocket)

**Функциональность:**
- Создание чата при обращении к продавцу по конкретному товару
- Мгновенная доставка сообщений через WebSocket
- Отслеживание прочитанности сообщений
- Push-уведомления в браузере (Web Notifications API)
- Пагинация истории сообщений
- Техподдержка через localStorage (без серверной части)

## 10. Ключевые паттерны и подходы

### 10.1. Feature-Sliced Design (упрощённый)
Проект частично следует Feature-Sliced Design:
- `_features/` — feature-специфичные модули (feed, category, ad, favorites, product)
- `_components/` — переиспользуемые компоненты уровня страницы
- `_lib/` — утилиты уровня модуля
- Подчёркивание в имени (`_`) обозначает приватные директории, исключённые из маршрутизации Next.js

### 10.2. Server Components + Client Components
- **Server Components** (по умолчанию): layout'ы, страницы товаров и продавцов — SSR с данными, загруженными на сервере
- **Client Components** (`"use client"`): интерактивные страницы — формы, чаты, админ-панель, компоненты с состоянием

### 10.3. SSR Prefetching + Hydration
- В корневом `layout.tsx` выполняется `prefetchQuery` для текущего пользователя
- `HydrationBoundary` с `dehydrate(queryClient)` передаёт серверные данные клиенту
- Это обеспечивает мгновенное отображение без «мигания» при загрузке

### 10.4. Оптимистичные обновления
- `LikeButton` обновляет состояние мгновенно, откатывая при ошибке

### 10.5. URL-state синхронизация
- Библиотека `nuqs` обеспечивает типобезопасную синхронизацию фильтров с URL query-параметрами
- Позволяет делиться ссылкой с примененными фильтрами

### 10.6. Централизованная обработка ошибок
- HTTP-клиент автоматически сбрасывает авторизацию при 401
- React Query с `retry: false` и кастомным `throwOnError`
- Глобальный `staleTime: 60s` для оптимизации запросов

## 12. Бизнес-процессы

### 12.1. Жизненный цикл объявления
1. Пользователь создаёт объявление (заполняет форму, выбирает категорию, загружает фото)
2. Объявление переходит в статус `MODERATE` (на модерации)
3. Администратор одобряет (`APPROVED`) или отклоняет (`DENIED`) с указанием причины
4. Одобренные объявления появляются в ленте и доступны для поиска
5. Владелец может скрыть (`isHide: true`) / показать / удалить / отредактировать объявление

### 12.2. Процесс покупки/коммуникации
1. Покупатель находит товар через поиск/каталог
2. Нажимает «Написать продавцу» → создаётся чат, привязанный к товару
3. Обмен сообщениями в реальном времени через WebSocket
4. Push-уведомления о новых сообщениях

### 12.3. Процесс пополнения баланса
1. Пользователь вводит сумму пополнения
2. Создаётся платёж → получение URL платёжной системы
3. Перенаправление на страницу оплаты
4. Автоматическая проверка статуса каждые 30 секунд
5. При успешной оплате → баланс обновляется

### 12.4. Процесс продвижения
1. Администратор создаёт типы продвижения (с ценой за день)
2. Пользователь выбирает тип продвижения и товар
3. Указывает количество дней → списание с баланса
4. Продвигаемые товары (`hasPromotion: true`) выделяются в ленте

### 12.5. Процесс размещения рекламного баннера
1. Пользователь подаёт заявку (изображение, URL, место, период)
2. Администратор создаёт/управляет баннерами
3. Баннеры отображаются в указанных местах (лента, профиль, чаты, избранное)
4. Система отслеживает просмотры (общие и уникальные)
5. Пользователь видит статистику своих баннеров

- Многоступенчатая Docker-сборка — минимальный размер образа
- Unoptimized images — делегирование оптимизации CDN/бэкенду

### 13.2. Безопасность
- Сессионная аутентификация через HttpOnly cookie (`session_id`)
- Middleware-защита на уровне маршрутов
- Серверная проверка прав администратора

### 13.5. Масштабируемость

- Независимые React Query хуки для каждой сущности

## 17. Резюме

«Торгуй сам» — это полнофункциональный веб-маркетплейс, построенный на современном стеке технологий (Next.js 16, React 19, TypeScript) с серверным и клиентским рендерингом. Приложение реализует классическую функциональность доски объявлений: размещение товаров с иерархической категоризацией, мощную систему поиска и фильтрации, real-time чат между покупателями и продавцами, систему отзывов, монетизацию через систему пополнения баланса и рекламные баннеры, а также полноценную административную панель для управления контентом и пользователями.

Проект контейнеризирован с помощью Docker, использует многоступенчатую сборку для оптимизации production-образа и развёртывается через Docker Compose. Frontend взаимодействует с отдельным REST API бэкендом и WebSocket-сервером для обеспечения реалтайм-коммуникаций.
